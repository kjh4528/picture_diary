## Phase 0: 환경 설정 및 폴더 구조 잡기

> **왜 먼저 해야 하나?**  
> 집을 짓기 전에 설계도와 터를 잡는 것과 같습니다. 폴더 구조와 환경이 틀리면 이후에 작성한 코드가 서로 연결되지 않고, API 키 노출 같은 보안 사고가 날 수 있습니다.

### 0-1. 폴더 골격 생성

```bash
mkdir -p picture_diary/agents picture_diary/domains picture_diary/outputs
cd picture_diary
touch agents/__init__.py
```

**꼭 알아야 할 것:**
- 폴더명은 반드시 `picture_diary` (하이픈 ❌, 밑줄 ✅)
- `agents/__init__.py`가 없으면 Python이 `agents`를 패키지로 인식 못함 → `from agents.scene import ...` 임포트 실패
- `outputs/`는 코드에서 `Path("outputs").mkdir(exist_ok=True)`로 자동 생성해도 됨

### 0-2. `.env` 파일 생성 (API 키 관리)

```
OPENAI_API_KEY=sk-...
FAL_KEY=...
```

**왜 `.env`를 쓰는가?**  
코드 파일에 API 키를 직접 쓰면, GitHub에 올렸을 때 전 세계에 키가 공개됩니다. 실제로 이런 실수로 하루 만에 수십만 원이 청구되는 사례가 있습니다. `.env` 파일은 코드와 분리된 "잠긴 서랍"입니다. Python에서는 `load_dotenv()` + `os.getenv()`로 읽습니다.

### 0-3. `.gitignore` 작성

```gitignore
.env
.venv/
outputs/
*.png
*.mp4
__pycache__/
*.pyc
```

**주의:** `.env`가 이 파일에 없으면 `git add .` 한 번에 API 키가 커밋될 수 있습니다. 제출 전 반드시 확인.

### 0-4. 패키지 설치 및 `requirements.txt` 생성

```bash
uv pip install openai fal-client python-dotenv requests
uv pip freeze > requirements.txt
```

**왜 `uv`를 쓰는가?**  
이 과정에서는 `uv`를 패키지 관리자로 사용합니다. `pip`보다 빠르고, 의존성 충돌을 잘 처리합니다.

### 0-5. `diary.md` 작성 (프로젝트 원료)

직접 쓴 일기 텍스트입니다. 나중에 `agents/scene.py`가 이 파일을 읽어 장면을 추출합니다. 300~500자, 화면으로 보이는 구체적인 장면이 있어야 합니다.

**완료 기준:** `ls` 했을 때 `.env`, `.gitignore`, `diary.md`, `agents/__init__.py`가 보이면 Phase 0 완료.

## Phase 1: 핵심 에이전트 모듈 개발

> **왜 이렇게 나누는가?**

### 1-1. `agents/scene.py` — 일기 텍스트 → 장면 JSON 추출

**무슨 역할?**  
OpenAI Chat API를 호출해서 일기 텍스트를 분석하고, 그림으로 그릴 수 있는 장면 목록을 JSON 형태로 자동 추출합니다.

**구현해야 할 함수:**

```python
def extract_scenes(diary_text: str) -> list[dict]:
    """
    입력: 일기 텍스트 문자열
    출력: 장면 딕셔너리 리스트
    각 딕셔너리에 반드시 포함:
      - scene_kr: 한국어 장면 설명
      - prompt_en: 영문 이미지 생성 프롬프트
    """
```

**핵심 기술 — Chat JSON 모드:**

```python
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user", "content": diary_text}
    ],
    response_format={"type": "json_object"},  # ← 핵심!
    temperature=0.7,
    max_tokens=1500,
)
scenes = json.loads(response.choices[0].message.content)["scenes"]
```

**왜 `response_format={"type": "json_object"}`를 쓰는가?**  
일반 Chat API는 자유로운 텍스트를 반환해서, JSON처럼 생겼지만 파싱이 실패할 수 있습니다. JSON 모드를 켜면 모델이 반드시 유효한 JSON을 반환하도록 강제됩니다. 채점기가 `scenes` 리스트를 파이썬 코드로 검증하기 때문에 이 부분이 매우 중요합니다.

**자주 발생하는 문제:**
- `KeyError: 'scenes'` → 시스템 프롬프트에 최상위 키를 `scenes`로 지정했는지 확인
- 장면이 4개가 아니라 3개나 5개 → 시스템 프롬프트에 "반드시 4개 장면"을 명시
- `prompt_en`이 한국어로 나옴 → 시스템 프롬프트에 "영어로 작성" 명시

**시스템 프롬프트 핵심 요소:**
```
1. 출력 JSON 스키마 명시 (scenes 배열, 각 항목 필드)
2. 장면 수 = 4개 고정
3. prompt_en은 영어, shot/angle/lighting 포함
4. 스타일 일관성 어휘 (예: watercolor illustration)
```

### 1-2. `agents/image.py` — 장면 프롬프트 → 이미지 파일

**무슨 역할?**  
`scene.py`가 만든 영문 프롬프트를 받아, DALL-E 또는 fal.ai FLUX 모델로 이미지를 생성하고 파일로 저장합니다.

**구현해야 할 함수:**

```python
def generate_image(
    prompt: str,
    output_path: str,
    model: str = "dalle"   # "dalle" 또는 "flux"
) -> str:
    """
    입력: 영문 프롬프트, 저장 경로, 모델 선택
    출력: 저장된 파일 경로 문자열
    """
```

**두 모델의 차이 — 꼭 알아야 할 포인트:**

| 항목 | DALL-E (OpenAI) | FLUX (fal.ai) |
|------|-----------------|---------------|
| 응답 방식 | base64 또는 URL | URL |
| 응답 구조 | `response.data[0].url` | `result["images"][0]["url"]` |
| 비용 | ~$0.04/장 | ~$0.003/장 |
| 속도 | 5~15초 | 2~5초 |
| API 키 변수명 | `OPENAI_API_KEY` | `FAL_KEY` |

**자주 발생하는 문제:**
- DALL-E 응답 구조를 FLUX에 그대로 쓰면 `AttributeError` 발생
- `outputs/` 폴더가 없으면 `FileNotFoundError` → `Path(output_path).parent.mkdir(exist_ok=True)` 필수
- 이미지 URL이 임시 URL이라 시간이 지나면 만료됨 → 받자마자 파일로 저장해야 함

### 1-3. `agents/video.py` — 이미지 → 영상 파일

**무슨 역할?**  
생성된 이미지를 입력으로 받아 짧은 영상(mp4)을 만드는 비동기 작업입니다. 영상 생성 API는 결과가 바로 나오지 않고, 완료될 때까지 주기적으로 상태를 확인해야 합니다(폴링).

**구현해야 할 함수:**

```python
def generate_video(
    image_path: str,
    output_path: str
) -> str:
    """
    입력: 입력 이미지 경로, 저장할 영상 경로
    출력: 저장된 .mp4 파일 경로 문자열
    """
```

**왜 비동기 폴링이 필요한가?**  
영상 생성은 이미지 생성보다 훨씬 오래 걸립니다(30초~수분). API에 요청을 보내면 즉시 작업 ID를 받고, 이후 완료 여부를 주기적으로 확인(폴링)하는 방식을 씁니다.

```python
# 폴링 패턴 예시
import time

response = api.submit_job(image_url)
job_id = response["job_id"]

while True:
    status = api.check_status(job_id)
    if status["state"] == "completed":
        video_url = status["video_url"]
        break
    time.sleep(3)  # 3초 대기 후 재확인
```

**주의:**
- 영상 생성 API는 과정마다 다를 수 있으니 강사 안내를 따릅니다
- 출력 파일은 반드시 `.mp4` 확장자여야 채점 통과

## Phase 2: 파이프라인 조립 — `pipeline.py`

> **왜 `pipeline.py`가 필요한가?**  
> 3개의 에이전트 모듈(scene, image, video)을 하나의 흐름으로 연결하는 지휘자 역할입니다. 사용자는 이 파일 하나만 실행하면 일기 텍스트에서 영상까지 자동으로 만들어집니다. "파이프라인"이라는 개념 자체가 이 프로젝트의 핵심 학습 목표입니다.

**구현해야 할 함수:**

```python
def picture_diary_pipeline(diary_text: str) -> dict:
    """
    전체 파이프라인 실행 함수
    입력: 일기 텍스트 문자열
    출력: {
        "scenes": [...],    ← 추출된 장면 리스트 (필수)
        "images": [...],    ← 생성된 이미지 경로 리스트 (필수)
        "videos": [...]     ← 생성된 영상 경로 리스트 (선택)
    }
    """
```

**파이프라인 흐름:**

```
diary_text
    │
    ▼
[1] extract_scenes(diary_text)
    │  → scenes: list[dict]
    ▼
[2] generate_image(scene["prompt_en"], output_path)  ← 각 장면마다
    │  → images: list[str]
    ▼
[3] generate_video(image_path, video_path)            ← 선택
    │  → videos: list[str]
    ▼
return {"scenes": ..., "images": ..., "videos": ...}
```

**실행 진입점 패턴:**

```python
if __name__ == "__main__":
    diary_text = Path("diary.md").read_text(encoding="utf-8")
    result = picture_diary_pipeline(diary_text)
    print(f"장면 수: {len(result['scenes'])}")
    print(f"이미지: {result['images']}")
```

**자주 발생하는 문제:**
- `from agents.scene import extract_scenes` → `agents/__init__.py`가 없으면 임포트 에러
- 반환값 딕셔너리에 `scenes`, `images` 키가 반드시 있어야 함 (채점 대상)
- 영상 생성은 시간이 오래 걸리므로 `videos` 키는 선택사항으로 처리 가능

| `product` | 제품 카탈로그 스타일 이미지 | `day5_01_product_catalog.py` |
| `emoji` | 이모티콘 캐릭터 시리즈 | `day5_02_emoticons.py` |

# 📚 실습 가이드 vs 로드맵 비교 분석

## 2-1. 실습 가이드 파일별 내용 요약

### 📄 `wf3_day1-self1_실습가이드.docx`
**핵심 주제:** OpenAI DALL-E 첫 호출 + 환경 설정

이 세션에서 처음으로 `picture_diary` 폴더를 만들고, `.env`에 API 키를 저장하는 방법을 배웁니다. 일기(`diary.md`)를 직접 쓴 뒤, `day1_self1.py` 파일에 4개의 함수(`load_api_key`, `build_scene_prompt`, `generate_image`, `save_image`)를 직접 채워 첫 번째 이미지(`outputs/scene01_dalle.png`)를 생성합니다.

**만드는 파일:** `.env`, `.gitignore`, `diary.md`, `day1_self1.py`, `dev_log.md`  
**사용 API:** OpenAI DALL-E (gpt-image-2)  
**핵심 학습:** load_dotenv(), base64 디코딩, `response.data[0].b64_json` 구조

### 📄 `wf3_day1-self2_실습가이드.docx`
**핵심 주제:** 같은 장면, 다른 프롬프트 — 시각 어휘 비교

Day1-self1에서 만든 이미지를 기반으로, **샷(WS/CU/low angle), 앵글, 조명**이라는 시각 어휘를 배웁니다. 같은 일기 장면을 3가지 프롬프트 변형으로 만들어 비교하고(`scene01_ws.png`, `scene01_cu.png`, `scene01_low.png`), Day 2 self1을 위한 씨앗 메모(`scene_draft_seed.md`)를 작성합니다.

**만드는 파일:** `day1_self2.py`, 이미지 3장, `scene_draft_seed.md`  
**핵심 학습:** 시각 어휘(shot/angle/lighting), try/except 안전 호출 패턴

### 📄 `wf3_day2-self1_실습가이드.docx`
**핵심 주제:** 4장면 초안 문서화 + 검증 스크립트

Day1의 씨앗 메모를 바탕으로, 그림일기 4장면을 정식 초안(`scene_draft.md`)으로 정리합니다. **API 호출 없이** 순수하게 문서 작업만 합니다. `day2_self1.py`는 `scene_draft.md`의 구조(4장면 × 7필드)를 검증하는 파이썬 스크립트로, 정규식으로 헤딩을 파싱하고 필드 존재 여부를 확인합니다.

**만드는 파일:** `scene_draft.md`, `day2_self1.py`  
**핵심 학습:** 5가지 시각 어휘(shot/angle/light/composition/lens), 정규식, 데이터 검증의 중요성

### 📄 `wf3_day2-self2_실습가이드.docx`
**핵심 주제:** JSON 스키마 설계 + fal.ai FLUX 첫 호출

`scene_draft.md`의 4장면을 `scene_prompts.json`으로 변환하고, **fal.ai의 FLUX-schnell 모델**로 첫 번째 장면 이미지를 생성합니다. DALL-E와 fal.ai의 응답 구조가 다르다는 것을 직접 체험합니다(`response.data[0].url` vs `result["images"][0]["url"]`). `FAL_KEY`를 `.env`에 추가하는 것도 이 세션입니다.

**만드는 파일:** `scene_prompts.json`, `day2_self2.py`, `outputs/scene01_fal.png`  
**핵심 학습:** JSON 스키마 설계, fal_client.subscribe(), 두 API 응답 구조 비교

### 📄 `wf3_day3-self1_실습가이드.docx`
**핵심 주제:** `agents/scene.py` 완성 — 일기 자동 장면 추출

Day 2까지는 사람이 직접 `scene_prompts.json`을 작성했습니다. 이제 그 작업을 **LLM이 자동으로** 하도록 `agents/scene.py`를 만듭니다. OpenAI Chat API의 JSON 모드를 사용해, 일기 텍스트를 입력받아 `scenes` JSON을 자동 추출합니다. `agents/__init__.py`도 이 세션에서 만듭니다.

**만드는 파일:** `agents/__init__.py`, `agents/scene.py`, `day3_self1.py`, `scene_extracted.json`  
**핵심 학습:** Chat JSON 모드, 시스템 프롬프트 설계, 함수 역할 분리(추출/검증/저장), 모듈화

`agents/scene.py`가 추출한 JSON을 입력으로 받아, **DALL-E 또는 FLUX 중 선택**해서 4장을 일괄 생성합니다. 모델 분기 함수(`generate_image`), 캐릭터 일관성을 위한 공통 스타일 어휘(`COMMON_STYLE`), `batch_generate` 함수를 구현합니다.

**핵심 학습:** 모델 분기(dalle/flux), 시드 고정으로 일관성 확보, batch_generate 패턴

## 2-2. 실습 가이드 → 로드맵 매핑표

| 실습 가이드 파일 | 로드맵 단계 | 설명 |
|----------------|------------|------|
| `day1-self1` | **Phase 0** (0-1 ~ 0-5) + Phase 1 preview | 폴더 생성, .env, .gitignore, diary.md 작성, 첫 DALL-E 호출 |
| `day1-self2` | **Phase 0** (심화) + Phase 1-2 준비 | 시각 어휘 학습, 프롬프트 변형 실험. `image.py`에 들어갈 시각 어휘 감각을 손으로 익히는 단계 |
| `day2-self1` | **Phase 1-1 준비** | `scene.py`의 출력 구조(4장면 × 필드)를 사람이 먼저 설계하고 검증. 자동화 전 수동 버전 |
| `day2-self2` | **Phase 1-2 준비** + Phase 1-1 심화 | JSON 스키마 확정, fal.ai 연결. `image.py`와 `scene.py`에 들어갈 API 구조를 직접 체험 |
| `day3-self1` | **Phase 1-1** (`agents/scene.py` 완성) | 로드맵의 핵심 에이전트 모듈 1번 완성 |
| `day3-self2` | **Phase 1-2** (`agents/image.py` 완성) | 로드맵의 핵심 에이전트 모듈 2번 완성 |
| *(Day 4~5 예상)* | **Phase 1-3** + Phase 2 + Phase 3 | `agents/video.py` 완성, `pipeline.py` 조립, 도메인 응용 |

## 🔑 로드맵 vs 실습 가이드: 가장 큰 차이점

**실습 가이드의 순서:**
```
Day1: 환경설정 → 직접 이미지 생성 (손으로 체험)
Day2: 문서화 → JSON 설계 → 다른 API 체험
Day3: agents/ 모듈 자동화 (코드로 추상화)
Day4~5: 파이프라인 조립 → 도메인 응용 (예상)
```

**로드맵의 순서 (결과물 중심):**
```
Phase 0: 환경설정
Phase 1: agents/ 3개 파일 완성
Phase 2: pipeline.py 조립
Phase 3: 도메인 응용
Phase 4: 마무리/제출
```

**핵심 차이:**  
실습 가이드는 **"왜 이 코드가 필요한지"를 몸으로 체험시키는 순서**로 짜여 있습니다. 즉, Day 1~2에서 직접 손으로 하던 작업(장면 작성, JSON 변환, API 호출)을 Day 3부터는 코드가 자동으로 하도록 추상화합니다. 로드맵은 **최종 제출 구조 기준으로** 거꾸로 역산한 순서입니다.

실습 가이드를 따라가면서 로드맵의 어느 단계를 쌓고 있는지 의식하면, 각 실습의 의미가 훨씬 잘 보입니다.
